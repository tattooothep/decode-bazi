## 0. สภาพปัจจุบัน (ประเมินจากไฟล์จริง)

### 0.1 Engine มีข้อมูลเยอะมากอยู่แล้ว — แต่ผู้ใช้ "มองไม่เห็น"

ไฟล์ `src/lib/astro/uranian/engine.ts` (`uranianChart()`) คำนวณ deterministic ครบ:

| ข้อมูลที่ engine ผลิต | field | ปริมาณ | ผู้ใช้เห็นตอนนี้ไหม |
|---|---|---|---|
| ดาว/จุด 10–12 ดวง + `dial90` (0–90°) | `points[].dial90` | 10 (ไม่มีเวลา) / 12 (มีเวลา) | ❌ เห็นแค่ AI เล่าเป็นข้อความ |
| จุดส่วนตัว ☉☽Asc MC ราหู จุดเมษ | `personalPoints[]` | 6 | ❌ |
| ครึ่งผลรวมทุกคู่ (Halbsumme) | `halbsummen[]` | สูงสุด 66 คู่ | ❌ (ไม่ส่งเข้า prompt ด้วยซ้ำ) |
| ภาพดาว (Planetenbild) ที่เข้า orb 1.5° | `planetaryPictures[]` | สูงสุด 60 | ⚠️ เห็นเป็น bullet ข้อความใน prompt |
| จุดไว (sensitive Punkte) sum/diff | `sensitivePoints[]` | สูงสุด 60 | ⚠️ เช่นเดียวกัน |
| ทรานส์เนปจูน Witte (ชื่อ+เจ้าราศี+แหล่ง) | `witteTransneptunians` | 4 (Cupido/Hades/Kronos/Zeus) | ⚠️ ยังไม่มีตำแหน่ง (เฟส 2) |

**ช่องว่างหลัก:** พลังทั้งหมดของยูเรเนียนคือ **"ภาพ" (Bild)** — ดาวหลายดวงมาเรียงซ้อนบนหน้าปัด 90° = จุดขายที่มองเห็นได้ทันที แต่ตอนนี้ผู้ใช้เห็นแค่ตัวหนังสือที่ AI เรียบเรียง (`renderUranianPrompt` ใน `src/lib/astro/uranian/render.ts` → ป้อน Claude → คืนข้อความล้วน ใน `public/master-fusion.html`). **ไม่มีวงล้อ/หน้าปัดภาพเลยแม้แต่หน้าเดียว.**

### 0.2 โครงสร้างการแสดงผลใน master-fusion.html

- `public/master-fusion.html` แสดงยูเรเนียนเป็น "แผงอ่าน" (text panel) เหมือนอีก 5 ศาสตร์ — แต่ละศาสตร์ AI อ่านแยก แล้ว Claude หลอมรวม (บรรทัด 1000, 1160–1162)
- ยูเรเนียนตั้ง `needsBirthTime:false`, `available:true`, `costYam:12` (สอดคล้อง `src/lib/fusion5/disciplines.ts:53`)
- ข้อมูล structured (dial90/pictures) **ไม่เคยถูกส่งถึง browser** — ถูกกินไปในชั้น prompt ฝั่ง server เท่านั้น
- ชั้นเวลา/กลุ่ม/คู่ ของยูเรเนียนยังปิด: `src/lib/fusion5/multi-year.ts:99` คืนค่าว่างสำหรับ uranian ("เฟส 1 = แผงอ่าน natal · ยังไม่ทำชั้นเวลาคู่/กลุ่ม")

### 0.3 ของเดิมที่ reuse ได้ทันที (สำคัญมากต่อ effort)

1. **วงล้อ SVG อ้างอิงพร้อมใช้** — `public/tianxing.html` มีวงล้อจักรราศีสมบูรณ์: helper `wpt(lon,R)` (polar→cartesian, บรรทัด 506), `arcPath()` (507), กันดาวชนกันด้วยชั้นรัศมี `RADII=[206,178,150]` (588), tooltip hover/tap/focus (`wireTooltips`, 631), เข็มทองลัคนา, glyph ราศี, ขีดองศา — สร้างเป็น string ล้วน ไม่พึ่ง lib ภายนอก (สอดกับ CSP) → **คัดลอกแนวได้ตรง ๆ**
2. **คณิตหน้าปัด 90° เขียนไว้แล้ว 2 ที่** — `midpointLon()` + `dial90Distance()` ทั้งใน `engine.ts` และ `day-sniper.ts:452/456` (นิยามเดียวกัน) → ตำแหน่งบนหน้าปัดคือ `dial90 * 4` (ขยาย 0–90 → 0–360 องศาวงกลม)
3. **"เข็ม D" ปลุกจุดคมตามวัน มีอยู่แล้ว** — `computeNeedleD()` ใน `src/lib/fusion5/day-sniper.ts` ยิงดาวจรเทียบจุดกึ่งกลาง natal บน dial 90° รายวัน (feature #7 แทบไม่ต้องสร้าง engine ใหม่)
4. **ชั้น Auslösung r389** — `src/lib/astro/uranian/auslosung.ts` มี transit/Sonnenbogen/progression → 5 มุมแข็ง พร้อมแปลง arc→อายุ→ค.ศ. (feature #7 ระดับลึก)
5. **คัมภีร์พร้อม** — `10-witte-canon-de.md` (verbatim DE, PD) + `11-method-reading-uranian.md` (พจนานุกรมภาพดาว 44KB, สำนวนระบบเรา) + `article-geometry.html` (หน้าบทความมีอยู่แล้ว) → feature #5/#6 คือการ "ต่อท่อ" ไม่ใช่สร้างเนื้อหาใหม่

## 3. เจาะลึก: "วงล้อ 90° interactive" ทำ SVG ยังไง (ฟีเจอร์ #1)

### 3.1 หลักคิดต่างจากวงล้อ 360° ยังไง

- วงล้อจักรราศีปกติ (tianxing) = 360° ราศีจริง
- **หน้าปัด 90° (dial 90°)** = พับ 4 ควอดแรนต์ทับกัน → ดาวที่อยู่ห่างกันพอดี 90°/180°/270° จะ **ทับกันบนหน้าปัด** = นี่คือ "ภาพดาว" · ดาวที่ทับกัน ≈ มีความสัมพันธ์มุมแข็ง → จุดขายคือ **"เห็นภาพดาวด้วยตา"**
- ตำแหน่งบนวงล้อ: ใช้ `p.dial90` (0–90 จาก engine) แปลงเป็นมุมวงกลม `angle = p.dial90 * 4` (0–360) → เต็มวง

### 3.2 โครงสร้าง SVG (อิงแนว tianxing.html — reuse ได้ตรง)

```
viewBox 0 0 640 640 · ศูนย์ (320,320)
helper: dpt(dial90, R) = polar→cartesian (คัดจาก wpt() tianxing:506)
        มุมจริง = dial90 * 4 องศา
วงแหวน:
  • วงนอก r=308 (ขอบหน้าปัด)
  • สเกล 0–90° · ขีดใหญ่ทุก 15° (= ครึ่งเหลี่ยม 45°/หาร) · ขีดเล็กทุก 1–5°
  • ป้ายเลข 0/15/30/45/60/75 รอบขอบ (บอกว่าเป็น "องศาหน้าปัด" ไม่ใช่ราศี)
  • วงในกลวง r=86 ใส่สรุป (ชื่อดวง/จำนวนภาพดาว)
```

### 3.3 การวางดาว + กันชน (คัด logic tianxing:586–619)

```
1. เรียงดาวตาม dial90
2. ดาวที่ dial90 ต่างกัน < ~3° = "ทับกัน" → ดันออกคนละชั้นรัศมี (RADII หลายระดับ)
   → การที่มันอยู่ชั้นเดียวกัน/ใกล้กัน = สื่อ "ภาพดาว" ให้ตาเห็น
3. แต่ละดาว: วงกลม core + glyph (☉☽☿♀♂♃♄⛢♆♇) + ขีดองศาจริงบนขอบ
4. เส้นเชื่อม (แกนสมมาตร): ลากเส้นบาง ๆ ระหว่างคู่ที่ engine ระบุเป็น planetaryPicture
   → เห็น "โครง" ของภาพดาว
```

### 3.4 Interactive (จุดขายคำว่า "หมุนได้")

- **หมุนหน้าปัด**: drag/ลากขอบ = หมุน offset ทั้งวง (ใช้ CSS `transform: rotate` บน `<g>` เดียว — ไม่ต้อง recompute) · หรือปุ่มเลื่อนไปยัง "จุดส่วนตัว" ให้มาอยู่ที่ 12 นาฬิกา (Witte เรียก "หมุนจุดที่สนใจขึ้นยอด")
- **แตะดาว→tooltip**: reuse `wireTooltips` pattern (hover/tap/focus, `#wtip`) · โชว์ชื่อไทย+เยอรมัน + dial90 + ภาพดาว/จุดไวที่เกี่ยว
- **แตะแกนสมมาตร**: โชว์สมการ A/B=C 3 ทิศ + คำอ่านจากพจนานุกรม (`11-method-reading-uranian.md`)
- **สลับ orb** (1.5° ↔ แคบ/กว้าง): ปุ่ม toggle → กรองภาพดาวที่โชว์ (engine ให้ orb จริงมาแล้ว ไม่ต้อง recompute แค่ filter ฝั่ง client)

### 3.5 SVG vs Canvas — เลือก SVG

| เกณฑ์ | SVG | Canvas |
|---|---|---|
| จำนวน element (10–12 ดาว + ขีด) | ✅ เบามาก | เกินจำเป็น |
| tooltip/แตะรายดาว | ✅ ต่อ event รายชิ้นได้เลย | ต้อง hit-test เอง |
| แนวที่มีอยู่ในโค้ดฐาน | ✅ tianxing/chart ใช้ SVG string | — |
| คมชัดทุกจอ (retina/ซูม) | ✅ vector | ต้อง scale DPR |
| แชร์/บันทึกภาพ | ✅ serialize → PNG ได้ | ได้ แต่ต้องเขียนเพิ่ม |

**สรุป: SVG** — ตรงกับ `tianxing.html`/`chart.html` ที่ทำสำเร็จแล้ว, เบา (element น้อย), ต่อ interactive ง่าย, และไม่ชน CSP (สร้าง string ใน page เดียว ไม่โหลด lib นอก)

### 3.6 Data flow ที่ยังขาด (สิ่งที่ต้องเปิด — แต่ **ไม่ทำในเอกสารนี้**)

ตอนนี้ `packet.ts` มี `points[].dial90`, `planetaryPictures[]`, `sensitivePoints[]`, `personalPoints[]` ครบ **แต่ไม่ถูกส่งถึง browser** (ถูกกินในชั้น `render.ts`→prompt) · ฟีเจอร์ #1 ต้องมี endpoint/ช่องส่ง structured uranian JSON ออกมาให้ client วาด (คล้าย `/api/chart` ส่ง analysis ให้ `chart.html`) — เป็นงาน Layer 3+4, additive, ไม่แตะ engine (Layer 0–2 คงเดิม)

## 4. หมายเหตุความปลอดภัย/สเกล (ตามกฎในหน่วยความจำ)

- **ไม่แตะ engine (Layer 0–2)** — `engine.ts`/`packet.ts`/`auslosung.ts` deterministic อยู่แล้ว · ทุกฟีเจอร์ข้างบนเป็น Layer 3 (ส่ง JSON) + Layer 4 (วาด/แสดง) additive
- **NO_PERCENT** — วงล้อ/การ์ดต้องไม่โชว์คะแนน/เปอร์เซ็นต์ทำนาย · orb = ระยะเชิงมุมจริง ใช้เรียง "ความคม" ไม่ใช่คะแนน (กฎข้อ 7 ใน `render.ts`)
- **ป้ายความซื่อสัตย์** — คำอ่านภาพดาวที่ไม่ใช่ Witte verbatim ต้องคง label `METHOD_READING_LABEL` (การอ่านเชิงวิธี · ไม่ใช่ถ้อยคำ Witte) — ฟีเจอร์ #2/#5/#6 ที่ดึงพจนานุกรมต้องพก label นี้ไปด้วย
- **scale 5,000 user** — ฟีเจอร์ #7 (ชั้นเวลา) คำนวณหนักสุด → ต้อง cache ต่อดวง (natal ไม่เปลี่ยน · `day-sniper` มี `MIDPOINT_CACHE` เป็นแบบอย่าง) + rate limit ตามกฎ
- **⛔ TNP ต้องห้าม** — ทุก UI ห้ามหลุด Apollon/Admetos/Vulkanus/Poseidon (Sieggrün ลิขสิทธิ์) · ใช้ได้เฉพาะ Cupido/Hades/Kronos/Zeus และเฟส 1 ยัง "ไม่มีตำแหน่ง" → วงล้อ **ห้ามวาดจุด TNP** จนกว่าจะ wire ephemeris เฟส 2

## 5. ยืนยัน: เอกสารนี้ "ไม่ได้บอกตรง"

- เอกสารนี้เป็น **การวิเคราะห์ช่องว่างเชิงสินค้า/UX เท่านั้น** — ไม่มีคำทำนาย ไม่ตีความดวงของใคร ไม่ให้คำฟันธง
- ไม่มีการอ้างถ้อยคำ Witte verbatim มาบอกว่าเป็นความจริงสำเร็จรูป · ทุกที่ที่พูดถึงคำอ่านภาพดาว ระบุชัดว่าเป็น "พจนานุกรมการอ่านเชิงวิธีของระบบ" (method reading) ไม่ใช่ verbatim
- ข้อเสนอทั้ง 7 เป็น "ตัวเลือกรอเจ้านายเคาะ" ตามกฎ **ถามก่อนแก้โค้ดทุกครั้ง** · **ไม่มีการแก้โค้ดใด ๆ** ในรอบนี้
- ทุกข้ออ้างอิงชี้ path ไฟล์จริงในโค้ดฐาน (ตรวจสอบย้อนได้) ตามกฎ **ห้ามโกหก พูดตรง — บอกที่มาข้อมูลชัดเจน**
