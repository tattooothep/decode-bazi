## Core Chain

1. **命宮 / 身宮**: capacity to carry technical complexity, decision pressure, repeated debugging, and long maintenance.
2. **兄弟**: product spec, code logic, prompt/API wording, documentation, integration contract, bug report clarity, and developer communication.
3. **官祿**: product direction, roadmap discipline, release standard, official platform promise, and credibility of the technical claim.
4. **僕役**: developers, vendors, maintainers, model providers, cloud partners, contractors, and whether work can be delegated.
5. **夫妻**: users, customers, app stores, marketplace/platform counterparties, API consumers, and integration acceptance.
6. **田宅**: server base, database, hosting, repository, data store, model knowledge base, and retained platform asset.
7. **財帛**: build cost, cloud bill, token cost, maintenance budget, and whether the product can pay for technical upkeep.
8. **疾厄 / 化忌 / 煞曜**: bugs, outage, data leak, security flaw, technical debt, vendor lock-in, unstable automation, or platform penalty.
9. **Timing**: 大限/流年/流月/流日 only when supplied.

## Product-Chain Rule

A feature is not a stable product. Require a chain:

- 兄弟 = specification, code, API, prompt, documentation,
- 官祿 = product direction and release standard,
- 僕役 = engineering/vendor/provider capacity,
- 夫妻 = user/platform/integration acceptance,
- 田宅 = infrastructure, database, repository, data asset,
- 財帛 = build cost and runway,
- 疾厄/化忌/煞曜 = bugs, outage, security leak, technical debt.

If 兄弟 is strong but 田宅/疾厄 are weak, the idea is clear but the system is unstable. If 官祿 is strong but 僕役 is weak, roadmap is good but engineering capacity is thin. If 夫妻 is strong but 財帛 is weak, users want it but infrastructure and maintenance cost can break the project.
